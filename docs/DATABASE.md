# Database

Local persistence is `drift` over SQLite (`lib/core/database/`). Backend (Postgres) mirroring `Document`, `Folder`, `Tag`, `UserSettings` is a Phase 2 deliverable — not implemented yet.

## Schema (v1 — Phase 0)

All tables live in `lib/core/database/tables/`. One table per file, matching spec Section 3:

| Table | File | Notes |
|---|---|---|
| `Documents` | `documents.dart` | `fileType`: `pdf`\|`image`. `syncStatus`: `none`\|`pending`\|`synced`\|`failed`\|`conflict` — unused until Phase 2 but in the schema now to avoid a breaking migration later. `ocrText` stays null until Phase 4. |
| `Pages` | `pages.dart` | Ordered pages belonging to a `Document`; `cropCoordinates` is JSON-encoded corner points. |
| `Folders` | `folders.dart` | Self-referencing (`parentId`) for nesting; `smartRule` (JSON) unused until Phase 3 smart folders. |
| `Tags` | `tags.dart` | Unique `name`. |
| `DocumentTags` | `document_tags.dart` | Many-to-many join — the spec's `Document.tags[]` is modeled relationally, not as an array/JSON column, so it stays queryable in SQLite. |
| `Signatures` | `signatures.dart` | `userId` nullable — Guest mode has no account. |
| `Stamps` | `stamps.dart` | Same nullable-`userId` rationale as `Signatures`. |
| `UserSettings` | `user_settings.dart` | Single-row table (`id` fixed at `0`). `serverConfig` from the spec is flattened into `server*` columns rather than nested JSON. `onboardingComplete` (added v2) gates the first-launch flow. |

`AppDatabase` (`database.dart`) registers all eight tables via `@DriftDatabase(...)`, opened through `drift_flutter`'s `driftDatabase(name: 'dokodocs_db')` helper. Exposed to the app via `databaseProvider` (`database_provider.dart`, plain Riverpod `Provider`).

## Code generation

Table/database code is generated by `drift_dev` via `build_runner`:
```
dart run build_runner build --delete-conflicting-outputs
```
This produces `database.g.dart` (git-ignored) — required before `flutter analyze`/`flutter test`/`flutter run` will succeed.

## Migration notes

- **v1 (Phase 0):** initial schema above, local-only.
- **v2 (Phase 1):** added `UserSettings.onboardingComplete` (bool, default `false`) — gates the first-launch onboarding flow. Migration: `m.addColumn(userSettings, userSettings.onboardingComplete)` in `AppDatabase.migration`.
- **v3 (batch/versioning/calendar/home pass):** the non-destructive-pipeline + favorites + version-history + dual-calendar + watermark upgrade. Changes:
  - `Pages.originalImagePath` (TEXT, **not null**) — the immutable capture. `localImagePath` is retained but repurposed as a cached processed preview/thumbnail; crop/filter/rotation are render-time metadata. Added via **raw SQL** (`ALTER TABLE pages ADD COLUMN ... TEXT NOT NULL DEFAULT ''`) rather than `m.addColumn`, because SQLite requires a default to add a NOT NULL column to a populated table and drift's `addColumn` can't emit one. **Backfilled** immediately: `UPDATE pages SET original_image_path = local_image_path` — every existing page becomes its own original, nothing overwritten (upholds the absolute non-destructive rule).
  - `Pages.needsReview` (bool, default `false`) — flags a page kept as a full frame due to low edge-detection confidence.
  - `Folders.isFavorite` (bool, default `false`) and `Folders.isDefault` (bool, default `false`) — favorite star + the undeletable auto-created "My Documents" folder.
  - `UserSettings.calendar` (`'ad'`|`'bs'`, default `'ad'`), `watermarkOnBatch` (default `true`), `watermarkOnSingle` (default `false`), `watermarkPosition` (`'bottom_right'`|`'top_right'`, default `'bottom_right'`).
  - New table **`DocumentVersions`** (`id`, `documentId`, `versionNumber`, `snapshotJson`, `snapshotPath` nullable, `changeLabel` nullable, `createdAt`) — page-metadata JSON snapshots, latest 10 per document kept. Created via `m.createTable(documentVersions)`.
- Every future schema change ships with: a bumped `schemaVersion`, a `MigrationStrategy` step in `AppDatabase`, and an entry appended to this file — no schema change should land without both.

## Repositories

Plain Dart wrapper classes in `lib/core/database/repositories/` (`DocumentsRepository`, `PagesRepository`, `FoldersRepository`, `UserSettingsRepository`) — no extra drift codegen needed beyond the tables themselves, since `AppDatabase`'s generated table getters (`db.documents`, `db.pages`, ...) already expose drift's fluent query builder. Each repository is exposed to the app via a Riverpod provider in `database_provider.dart`.
